### 1. 데이터 매핑 및 웹사이트 DB 구조 정의 (Data Mapping & DB Structure)

웹사이트의 핵심 비즈니스 지표(KPI) 추적을 위해 아래와 같이 필수 데이터 필드와 관계를 정의합니다. 모든 데이터는 **정확성과 실시간성**을 최우선으로 하여 설계되어야 합니다.

| 테이블/엔티티 | 핵심 데이터 필드 (Field Name) | 데이터 타입 | 설명 및 비즈니스 의미 | 연관 KPI |
| :--- | :--- | :--- | :--- | :--- |
| **User** | `user_id` (PK) | UUID | 사용자 고유 식별자 | N/A |
| | `acquisition_date` | DATETIME | 최초 유입 일시 | CAC 계산 기준 |
| | `lifetime_value` | DECIMAL(10, 2) | 고객 생애 가치 (LTV) | LTV/CAC 핵심 지표 |
| **Order** | `order_id` (PK) | UUID | 주문 고유 식별자 | N/A |
| | `user_id` (FK) | UUID | 주문한 사용자 ID | N/A |
| | `amount` | DECIMAL(10, 2) | 실제 거래 금액 | 매출 기여도 |
| | `conversion_date` | DATETIME | 구매 완료 일시 | 전환율 분석 |
| **Module** | `module_id` (PK) | INT | 판매된 모듈/상품 ID | N/A |
| | `price` | DECIMAL(10, 2) | 상품 기본 가격 | 수익성 분석 |
| | `status` | ENUM | 활성화/비활성화 상태 | 매출 추적 |

**DB 설계 원칙:**
1.  **관계 중심 설계:** User와 Order 테이블 간의 관계는 명확히 정의되어야 합니다.
2.  **수익 흐름 추적:** 모든 거래(Order)가 사용자(User)와 연결되어 LTV 계산의 기반이 됩니다.

### 2. 핵심 KPI 계산 로직 (LTV/CAC 중심)

ROI 극대화를 위해 다음 두 가지 핵심 지표의 정확한 계산 로직을 구현해야 합니다.

#### A. 고객 생애 가치 (LTV: Lifetime Value) 계산
**정의:** 특정 사용자가 우리 플랫폼에서 발생하는 총 예상 수익.
$$\text{LTV} = \frac{\text{평균 주문 금액} \times \text{평균 구매 주기}}{\text{고객 유지율}(\text{Retention Rate})}$$

*   **실행 로직:** `Order` 테이블의 데이터를 기반으로, 특정 사용자가 등록한 시점부터 현재까지 발생시킨 총 수익을 집계하고, 예상 유지율(다음 구매 예측) 모델을 적용하여 산출해야 합니다. 초기에는 **평균 주문 금액 (AOV)**과 **고객 이탈률**에 초점을 맞춰 구현합니다.

#### B. 고객 획득 비용 (CAC: Customer Acquisition Cost) 계산
**정의:** 신규 고객 1명을 확보하는 데 소요된 총 마케팅 및 운영 비용.
$$\text{CAC} = \frac{\text{총 마케팅 및 광고 비용}}{\text{신규 유입 사용자 수}}$$

*   **실행 로직:** `Order` 테이블과 연동하여, 특정 기간 내에 신규로 등록된 사용자 수와 해당 기간 동안 발생한 총 마케팅 지출액을 명확히 분리하여 집계해야 합니다. **모든 비용 항목은 추적 가능하도록 태깅되어야 합니다.**

#### C. 최종 ROI 지표
$$\text{LTV/CAC Ratio} = \frac{\text{LTV}}{\text{CAC}}$$

*   **최소 목표:** 이 비율이 **3.0 이상**임을 시각적으로 최우선으로 제시할 수 있도록 계산 로직의 정확성을 확보해야 합니다.

### 3. 외부 API 연동 및 인증 세부 사항 (API Integration & Authentication)

외부 데이터 소스(예: 광고 플랫폼, 결제 시스템 등) 연동을 위한 보안 및 접근 권한 세부 사항은 다음과 같습니다.

1.  **OAuth 2.0 (권장):**
    *   사용자 데이터 연동 또는 외부 서비스 통합 시 **OAuth 2.0 플로우**를 사용하여 사용자 동의 기반으로 인증을 수행해야 합니다. 이는 개인정보 보호 및 보안 표준을 충족합니다.
    *   **필요 사항:** `client_id`, `client_secret` 관리 및 토큰 갱신 메커니즘 구현에 집중하십시오.

2.  **API Key (보조):**
    *   특정 외부 서비스(예: 광고 성과 데이터) 연동 시, 해당 서비스에서 발급한 **API Key**를 사용하여 인증을 수행합니다.
    *   **보안 처리:** 모든 API Key는 환경 변수(`Environment Variables`) 또는 안전한 Secret 관리 시스템에 저장되어야 하며, 코드 내부에 하드코딩되어서는 안 됩니다.

3.  **인증 전략 요약:**
    *   **사용자 데이터 접근:** OAuth 2.0 기반의 사용자 동의 절차를 통해 진행.
    *   **외부 서비스 연동 (성과 데이터 등):** API Key를 통한 토큰 기반 인증을 수행하며, 키 관리는 최우선 보안 사항으로 처리.
